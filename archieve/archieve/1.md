# Writing Agent Design Principles

## Overview
[相同的事只做一次]
A writing agent helps an author shape their document for its reader. To design an effective writing agent, this specification defines the constraints, outlines a six-step procedure, and details the design for each step.


## Constraints

The writing agent is fundamentally constrained by six obvious facts:

1. LLM output depends on input context.
2. LLM context and attention are limited.
3. LLM can be confidently wrong.
4. Human attention is scarce.
5. Human intent may be vague or still forming.
6. Blind spots cannot be self-detected.

## Procedure

Procedure are design in respond to these constraints:

1. Gather minimal necessary context: intent, reader, domain, style. [#1, #2]
2. Structure ideas when intent is unclear. [#5]
3. Verify factual claims against external sources. [#3]
4. Simulate the target reader's experience. [#6]
5. Present changes for thorough yet efficient user review. [#3, #4]
6. Learn from user decisions. [#1]

gather → structure → verify → simulate → present → learn

## 5. Review

Goals:
1. User evaluates each edit in clean document context.
2. User decides edits thoroughly and efficiently.
3. Agent learns from resolved edits without ambiguity.

Design:
- Document remains clean text at all times. (1)
- Agent proposes edits grouped by issue in a separate interface. (2, 3)
- Default choice per edit is configurable: all old, all new, or subagent recommendation. (2)
- For each edit:
  - User chooses old or new. Choice syncs to document immediately. (1)
  - User can also request adjustment or explanation. (2)
  - Adjustment: agent proposes a replacement in the same group. (2, 3)
  - Explanation: agent responds in chat. (2)
- User resolves a group when satisfied. Resolved groups archive from the interface. (2)
- Superseded edits (direct document edit overlaps a pending edit) record no outcome. (3)
- Resolved edits form an ordered sequence of old/new choices. Agent learns from this sequence. (3)

## 6. Memory

Goals:
1. Agent produces better edits over time by learning from user decisions.
2. User has full visibility and control over what agent has learned.
3. Memory context cost stays bounded. (#2)

Design:

Three categories:
- **Knowledge** — domain facts not stated in the document. Agent records from user explanations in chat.
- **Principle** — writing rules as pluggable markdown files. Pre-installed from reference files, or recorded when user explicitly states a new rule.
- **Example** — resolved edit instances, classified under corresponding principles.

Three-layer retrieval:
- **Core principles** (narrative theory's four dimensions: care, understand, convinced, effortlessly) are always loaded. (3)
- **Operational guidelines** (word choice, section conventions, sentence construction, user-stated rules) are loaded by current task relevance. (3)
- **Examples** are loaded on demand, following the operational guideline currently active. (3)

Agent role:
- Agent records knowledge and classifies examples. Agent does not create, modify, or merge principles. (Constraint #3: classification is low-risk; modifying principles would amplify errors across all future edits.)
- On batch resolve, agent selects meaningful old/new choices and classifies each under a principle.
- Examples that do not fit any existing principle go into an uncategorized area. Agent notifies user when uncategorized examples accumulate.
- Examples that conflict with their principle are recorded as counterexamples under that principle. Agent notifies user when counterexamples accumulate.
- Agent notifies user of what it learned. No confirmation required.

User role:
- User can review, edit, or delete any memory entry at any time.
- Only user creates new principles or modifies existing ones.

Storage:
- All memory is markdown, exportable, document-scoped.

### Future Directions

The current design prioritizes simplicity. The following directions are deferred until real usage reveals concrete limitations.

**1. Principle DAG.** The three-layer retrieval is a pragmatic approximation. The underlying theoretical model is a directed acyclic graph: root nodes are the four core dimensions; each operational guideline is a derived node connected by an inference edge (parent principle + observable fact → derived guideline). Benefits: conflict resolution becomes traceable through root dimensions, retrieval becomes precise by walking the DAG from task-relevant roots. Deferred until three-layer shows concrete limitations (e.g., conflict resolution failures, imprecise retrieval).

**2. Activation model.** Instead of loading principles by task type (agent predicts relevance), principles are activated by evidence: an example from a resolved edit traces back to a principle, that principle becomes active for subsequent sessions. First session loads only core principles; operational guidelines activate as examples accumulate. Long-inactive principles decay back to standby. Benefits: context is data-driven rather than agent-predicted, avoids Constraint #3 in retrieval decisions. Deferred until attention dilution from static three-layer loading proves to be a real problem.

**3. Formal engine + LLM complementary architecture.** LLM and formal reasoning have complementary strengths. LLM handles semantics: interpreting user intent, judging text quality, reasoning in ambiguous contexts. Formal engine handles logic: consistency checking, conflict detection, dependency tracing across a rule set. Neither does the other's job well (Constraint #3: LLM fails at precise logic; formal engines cannot handle semantic ambiguity). Combined, a feedback loop emerges: LLM proposes new rules from user decisions → formal engine checks consistency against existing rules → on conflict, engine pinpoints the exact collision → LLM interprets the conflict semantically and suggests resolution → user decides → engine updates. Each step stays within its component's capability boundary. The Principle DAG (direction 1) is a natural substrate for this engine (e.g., dependency resolution via PubGrub-style algorithms). Deferred until the rule set grows complex enough that human management becomes impractical — the engine's value scales with rule count and interdependency.
